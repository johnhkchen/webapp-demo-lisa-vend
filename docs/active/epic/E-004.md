## Intent — the bigger-picture play

The Tailwind anchor is wired but the theme is bare — there is no shared vocabulary for the neon/glass look the demo is supposed to sell. This epic authors that design system as pure theme configuration: per-tetromino neon color tokens (one per piece so I/O/T/S/Z/J/L each own a signature hue), glassmorphic panel utilities (blur/translucency/border for scoreboard and preview surfaces), glow/shadow utilities for the neon bloom, row-clear flash keyframes, and 60fps-friendly transition classes. It is the styling contract the Board/Cell/panel components consume later — define the vocabulary once, centrally, so rendering work applies classes instead of hand-rolling ad-hoc CSS. P2 (the wow) is the whole point of this demo, and this is the reusable foundation everything visual leans on.

## Value to the design

Realizes the visual-impact foundation (P2): a single, central, reusable set of tokens/utilities/keyframes that makes the neon/glass theme consistent and cheap to apply, so later rendering tracks look finished and on-brand without re-deriving colors, glows, or flash timing per component.

## Done looks like

The Tailwind theme exposes per-tetromino neon color tokens, glassmorphic panel utilities, glow/shadow utilities, row-clear flash keyframes, and 60fps transition classes; each is referenceable by class/token name and renders its intended neon/glass/glow/flash effect in a throwaway probe; and no component or game-loop file was edited to achieve it — the vocabulary exists purely as theme config ready for components to consume.

## Context & constraints

Grounds against the current repo: Tailwind is already wired end-to-end via the @theme token bridge (commit 2203afd), which is this epic's SOLE prerequisite and already satisfied — the anchor exists, so this epic only extends it. Scope is strictly the styling vocabulary: tokens, utilities, keyframes, transition classes. HARD BOUNDARY (not hidden scope): this epic defines classes/tokens ONLY and edits no component (components/) or game-loop (lib/, app tick) files — applying these classes to Board/Cell/panels is the consuming rendering epic's work (E-003 seam and beyond), not this one. OUT OF SCOPE / separate concerns: actual line-clear animation wiring and rAF-driven juice (that consumes these keyframes but lives in the render/loop epic); scoring/preview/hold component markup (P4); Vercel deploy (P3). NON-GOALS respected: no backend, no accounts, no dependencies beyond the existing Tailwind stack — pure client-side theme config. Right-sized as ~one 2-hour block: a cohesive set of theme extensions over one clean file boundary (the Tailwind theme/globals) that DecomposeEpic can split into parallel token/utility/keyframe tracks. Permanent (reusable): unlike a single-use render wiring, this vocabulary is applied repeatedly by every visual surface for the life of the demo.
