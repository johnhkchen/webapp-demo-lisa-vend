## Intent — the bigger-picture play

Turn the invisible rules engine into a game a person can play. Someone wants the Board and Cell React components that render live board state as the CSS/DOM grid the scaffold settled on; a requestAnimationFrame tick that drives gravity in real time (piece falls, locks on landing, lines clear, game-over fires); and keyboard controls that map to move left/right, rotate, soft-drop, and hard-drop. This is the load-bearing seam between the two parallel tracks — E-002's pure, framework-free core supplies the next-state semantics, and this epic drives it from a React game-loop and forwards player input into it, holding rendering and rules on separable files exactly as CLAUDE.md mandates. It is the keystone for P1 playability: the rules already exist and the app already boots, but nothing on screen responds to a player until this loop is wired.

## Value to the design

Realizes Playability (P1), the top charter invariant and the core feature — spawn→move→rotate→drop→lock→clear→game-over becomes a thing you do with your hands, not a thing a test asserts. It is the single epic that converts a runnable skeleton plus a correct engine into an actual demoable game, and by the tie-breaker it is the largest jump in 'demoable sooner' available on the board.

## Done looks like

Loading the app in a browser shows a live board; a piece spawns and falls under gravity on a requestAnimationFrame tick; the keyboard moves, rotates, soft-drops, and hard-drops it; landed pieces lock, full lines clear, and the game ends on top-out — a stranger can play one continuous game from first spawn to game-over using only the keyboard, with no game-rule logic left to write in the render layer.

## Context & constraints

Grounds against a repo with no src/** modules yet on the board snapshot but two open keystone epics upstream: E-001 (scaffold-runnable-app-skeleton — settles Next.js App Router, Tailwind, and the DOM/CSS-grid-of-cell-divs board renderer) and E-002 (pure-game-core-engine — the framework-free lib/ API for tetrominoes, SRS rotation, collision, gravity/lock, line-clear, scoring, seeded RNG). Both are PREREQUISITES: this epic consumes E-002's next-state API and renders into E-001's grid; it does not re-implement game rules (that logic stays pure in lib/) and does not restyle the board (the neon/glass wow is P2, a separate epic). Scope is the render+loop+input seam only. Out of scope and to be left for later Feel (P4) epics: scoring UI, levels/speed curve, next-piece preview, hold, ghost piece, and localStorage high-scores — name-check them so they are not smuggled in here. In-bounds per the vision (single-player web Tetris, client-side only; no backend/persistence). Right-sized as one ~2-hour feature block; DecomposeEpic will split the render components, the rAF gravity loop, and the keyboard-input binding into their own stories/tickets on clean file boundaries.
