## Intent — the bigger-picture play

The core loop plays but feels utilitarian: pieces fall and clear with no aiming aid, no recovery from a bad drop, no preview of what's coming, no way to breathe, and clears happen with a silent instant redraw. Close that feel gap by adding the standard-Tetris comfort layer — a ghost piece (translucent landing projection of the active piece), a hold queue (swap active↔hold once per drop, C key), an exposed next-piece preview rendering the 7-bag lookahead, and a pause overlay (P key) — plus the line-clear juice that carries the wow: a row-clear flash/animation and smooth 60fps cell transitions. Extend the existing typed pure core (state/reducer/bag) with the new logic (ghost projection, hold slot + once-per-drop lock, exposed queue) and thread it through the React layer (Board, Cell, GameContainer, useGame, GameOverlay), keeping game rules pure and framework-free and animation concerns in the components. This is the single most visible batch of player polish still outstanding after core playability and deploy landed.

## Value to the design

Advances P4 (feel) — ghost, hold, next-preview and pause are exactly the descending-payoff touches the charter names that make the game satisfying to keep playing — and P2 (the wow) — line-clear flash and 60fps cell transitions are the on-screen juice that reads as 'finished'. It also demonstrates the pure-core discipline paying off: new mechanics land as reducer/state extensions behind clean file boundaries rather than as rendering hacks, keeping logic and juice separable and each track buildable in parallel.

## Done looks like

Playing the deployed game, a translucent ghost shows where the active piece will land and tracks it as you move/rotate; pressing C swaps the active piece into a hold slot (blocked to once per drop until the next lock); the upcoming 7-bag pieces are visibly rendered in a next-preview; P toggles a pause overlay that halts the loop and resumes cleanly; and clearing lines plays a visible row flash/animation with cells transitioning smoothly at 60fps rather than snapping. Core playability, scoring, and the production build stay green. Observable at the epic level: a bystander watching the screen sees a game that looks and feels shipped, not a prototype.

## Context & constraints

Grounds against the target layout in CLAUDE.md — pure logic lives in lib/ (state, reducer, bag) and rendering in components/ (Board, Cell, GameContainer, useGame, GameOverlay); the game core, 7-bag RNG, scoring, and Cloudflare/vinext deploy already landed (E-001…E-006), so this builds on a working, deployed loop. Non-goals respected: no backend, persistence beyond localStorage, auth, or analytics (N1–N4); all state stays client-side in React + rAF. PREREQUISITE (named, not hidden scope): the 7-bag generator must already expose lookahead beyond the current piece for the next-preview to render truthfully — if it only yields one piece at a time, exposing the queue is a required precursor within this epic's core work, not new demand. RIGHT-SIZING (PE-7) — this is at or above the upper bound of one ~2-hour block: it bundles six distinct player-facing features (ghost, hold, next-preview, pause overlay, line-clear animation, 60fps transitions). DecomposeEpic should expect this to plausibly split into TWO epics — a 'controls/overlay feel pack' (ghost + hold + next-preview + pause, mostly pure-core + component wiring) and a 'line-clear juice' epic (flash/animation + transition timing, mostly rendering/animation) — since the animation work has a different risk profile (frame timing, CSS/canvas transitions) than the deterministic reducer work; do not force all six into one story chain if it strains the graph or the 2-hour envelope.
