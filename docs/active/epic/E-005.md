## Intent — the bigger-picture play

Stand up the deploy pipeline early instead of at the finish line: add the Vercel project configuration, confirm the production build (`npm run build`) passes as a deploy gate, and connect Git integration so every push to the main branch auto-deploys and refreshes one shareable public URL. The play is to make shippability a standing, always-on property rather than a one-time end-of-project scramble — once wired, the demo has a real address the moment the skeleton runs, and each subsequent epic (game loop, neon/glass wow, feel) shows up live on that URL without any manual publish step. This front-loads the P3 invariant so 'a game nobody can reach' is never the demo's state, and the improving live link becomes a continuous proof of progress.

## Value to the design

Realizes Shippability (P3) as a continuously-satisfied capability: a passing production build enforced as a deploy gate plus Git auto-deploy means the charter's 'clean Vercel deploy with a shareable public URL' is true early and stays true on every push, converting build-passing (already implied by E-001) into an actually-reachable, self-updating demo — the reusable delivery substrate every later visual/gameplay epic ships through.

## Done looks like

Pushing a commit to the tracked branch triggers a Vercel build that runs the production build gate and, on success, publishes to a stable shareable public URL that serves the current app; the URL updates automatically on subsequent pushes with no manual deploy step; a failing production build blocks the deploy rather than shipping a broken page. Observable at the epic level: the running app is reachable at a public URL that tracks main.

## Context & constraints

Grounds against the current repo: the Next.js/TypeScript skeleton (E-001) with a passing `npm run build` and zero-warning lint gate (commits c2008c9, d68c42c) is the SOLE prerequisite — name it as a prerequisite, not hidden scope; this epic wires deploy on top of that runnable skeleton and does not create it. HARD BOUNDARY: this epic is self-contained deploy/CI wiring only — it touches Vercel project config, build/deploy settings, and Git integration; it edits NO game logic (`lib/`), NO components (`components/`), and NO game-loop/app-tick code. OUT OF SCOPE / separate concerns: the styling vocabulary (E-004), the game loop and rendering (E-002/E-003), and any gameplay/feel work — those ship THROUGH this pipeline but are not defined here. NON-GOALS respected (N1–N4): no backend, no accounts, no analytics, no native shell; this is client-only static/SSR deploy of a single-player web app. Right-sized as ~one 2-hour block over a clean config-only file boundary (Vercel/CI settings), which DecomposeEpic can split into project-config, build-gate, and Git-auto-deploy tracks. Permanent (reusable): the pipeline is a standing capability re-exercised on every push for the life of the demo, not single-use setup. Note: the actual `vercel --prod` / Git-integration connection may require human-held Vercel account credentials — flag that as an external prerequisite for the wiring step, not as in-scope engineering.
