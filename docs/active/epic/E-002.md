## Intent — the bigger-picture play

Build the pure, deterministic heart of the game in lib/ — the model of what Tetris *is*, independent of how it looks. Someone wants a self-contained rules engine that, given a board state and an input, returns the next board state: the 7-piece tetromino set with SRS rotation (including wall kicks), collision against walls and settled cells, gravity with lock-on-landing, line-clear detection, standard line-based scoring, and a seeded RNG so runs are reproducible. This is the substrate the render/loop track will drive; keeping it framework-free is what lets the logic track and the rendering track advance in parallel without colliding on the same files. It is the keystone for P1: no game loop, controls, or visual juice epic can produce a *playable* demo until these rules exist and behave correctly.

## Value to the design

Realizes Playability (P1), the top charter invariant and the core feature — a working game needs correct spawn/move/rotate/drop/lock/clear semantics before any of it can be wired to a screen. As pure TypeScript with no React/Next imports, it is directly unit-testable and honors the CLAUDE.md mandate that lib/ stay pure so rendering and logic tracks don't collide, unblocking parallel downstream work.

## Done looks like

lib/ exposes a framework-free (no React/Next imports) game-core API that, driven by a test or a thin harness, correctly: spawns all 7 tetrominoes, rotates them via SRS with wall kicks, blocks illegal moves via collision, applies gravity and locks a piece on landing, detects and clears full lines, updates a line-based score, and produces a reproducible piece sequence from a given seed. Given the same seed and input sequence, two runs yield identical board states. The rendering/loop epic can consume this API with no game-rule logic left to write.
