The blueprint. Which files change, the shape of each change, interfaces preserved, and ordering.
No code — the exact edits are specified in Plan/Implement. Follows Design decision B+A: prove the

| `app/globals.css` | modify | Remove the redundant raw `background`/`color` declarations from the `body` rule; keep `:root` vars, `@theme inline`, heights, font stack. Tighten the doc comment to state the bridge is now consumed via utilities. |

## File-level detail

### `app/globals.css` (modify)

Current shape (preserve unless noted):
```
@import "tailwindcss";
/* doc comment */
:root { --background: #0a0a0f; --foreground: #ededf2; }      ← keep (source of truth)
@theme inline { --color-background: var(--background);        ← keep (the bridge)
                --color-foreground: var(--foreground); }
html, body { height: 100%; }                                 ← keep
body { background: var(--background);   ← REMOVE this line
       color: var(--foreground);        ← REMOVE this line
       font-family: system-ui, …; }     ← keep the font-family
```

After change, the `body` rule retains only `font-family` (and `html, body { height: 100% }`
stays). The page-wide background/foreground colors now come from the `bg-background` /
`text-foreground` **utilities** applied in `layout.tsx`, which resolve — through `@theme inline`
— back to the same `:root` vars. Rendered output is identical.

- Update the doc comment so it no longer implies the body sets colors directly; note the colors
  flow token → utility.
- **Invariant preserved:** the dark canvas (`#0a0a0f` bg, `#ededf2` fg) is unchanged; only the
  *mechanism* moves from raw CSS to the Tailwind bridge.

### `app/layout.tsx` (modify)

Current `<body>`:
```
<body className="min-h-full flex flex-col">
```
After:
```
<body className="min-h-full flex flex-col bg-background text-foreground">
```
- No change to `<html>`, imports, `metadata`, or component structure.
- `bg-background`/`text-foreground` are the utilities generated by the `@theme inline` block —
  this is what pulls them into Tailwind's used-class set so they emit.

## Interfaces / contracts preserved

  canvas. No visual regression.
- **`@/*` alias, component tree, `lib/` boundary**: untouched. This ticket does not touch
  `components/` or `lib/` — no collision surface with sibling ticket T-001-02-02.
- **Build/lint scripts**: unchanged; must still pass.
- **No new files in `public/`, `components/`, `lib/`.**

## Ordering of changes

1. Edit `app/layout.tsx` to consume the utilities (makes them "used" so Tailwind emits them).
2. Edit `app/globals.css` to drop the now-redundant raw color rules + refresh the comment.
   (Order 1→2 keeps the page correctly colored at every intermediate state: after step 1 the
   body has both the utilities *and* the raw rules — same color, no flash — then step 2 removes
   the duplicate. Doing 2 before 1 would momentarily leave the body uncolored.)
3. Verify: `build` → grep emitted CSS for the bridge utilities/token → `dev` + `curl` → `lint`.
4. Commit as a single atomic change.

## Risk surface

- **Very low.** Two files, additive class list + deletion of two redundant CSS lines. The only
  failure modes: (a) a typo in the utility name (caught by grep/visual check), or (b) Tailwind
  content detection not picking up `layout.tsx` (contradicted by research — layout utilities
  already emit today). Both are covered by the Plan's verification steps.
