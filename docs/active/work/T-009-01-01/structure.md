# T-009-01-01 — Structure: vendor-b28-clay-kit-via-justfile

## Files created

### `justfile` (repo root)

New file. Two recipes:

- `default` — `@just --list`, so bare `just` is self-documenting instead of erroring.
- `sync-kit` — `mkdir -p styles/vendor` then `curl -fsSL <kit-url> -o styles/vendor/b28-clay.css`
  then an echo confirming the write.

Public interface this establishes for the rest of the repo/epic: `just sync-kit` is now the
canonical way to refresh the vendored kit. No other recipe needed by this ticket's acceptance
criterion.

### `styles/vendor/b28-clay.css` (new directory + new file)

Generated by running `sync-kit`, not hand-written. Byte-for-byte copy of
`https://b28.dev/kit/b28-clay.css` at fetch time. This is the artifact the acceptance criterion
checks for ("present on disk with real kit token content").

This path is the contract handed to `T-009-01-02` (`app/globals.css` will `@import` it). No other
file in this ticket references it — the import wiring is explicitly out of scope here.

### `styles/vendor/.gitkeep` — not needed

Once `sync-kit` runs, `styles/vendor/` is non-empty (contains `b28-clay.css`), so no placeholder
file is required to make the directory exist in git. `styles/` itself only exists because
`vendor/` exists under it — no separate creation step needed beyond `mkdir -p` in the recipe.

## Ordering / dependencies

Single-step change — no ordering concerns between files:

1. Write `justfile`.
2. Run `just sync-kit` once during Implement to produce `styles/vendor/b28-clay.css` and prove the
   recipe works end-to-end (not just that the `justfile` text looks right).
3. Confirm both are tracked by `git status` (new files, not ignored).

## Module boundaries

This ticket introduces exactly one new boundary: **`styles/` as a top-level directory for vendored
/ third-party CSS**, parallel to `app/`, `components/`, `lib/`. Nothing under `styles/vendor/`
should be hand-edited — it's fetch-generated output, analogous to `wrangler.jsonc`'s "generated,
committed verbatim" status described in `docs/deploy.md`. Future re-syncs overwrite it wholesale;
no per-line hand customization should ever live in `styles/vendor/b28-clay.css` (if the app needs
overrides on top of kit tokens, those belong in `app/globals.css`, layered on top — that's
literally `T-009-01-02`'s job, not this ticket's).

## No test-code changes

This ticket has no `lib/` or component logic, so there's nothing here for `vitest`. Verification
is procedural (run the command, inspect the file), covered in Plan below — not a unit test.
