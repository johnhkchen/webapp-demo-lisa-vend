## Test coverage

No unit tests were added or are needed ‚Äî this ticket has zero `lib/`/component logic to unit-test;
its correctness is procedural (the recipe either fetches and writes the file correctly, or it
doesn't). Verification was end-to-end: ran the actual command against the live URL twice, inspected
the resulting file's size and content by hand/grep.

As a regression check, ran the existing suites unmodified:
- `npm run test` ‚Äî 32 test files, 302 tests, all passing.
- `npm run build` (`vinext build`) ‚Äî succeeds cleanly. Expected, since the vendored CSS isn't
  imported by anything yet (that's `T-009-01-02`'s job) ‚Äî it can't perturb the build either way.

**Gap, by design, not oversight:** there is no automated test that `styles/vendor/b28-clay.css`
stays in sync with the upstream kit over time, or that `sync-kit` keeps working if `b28.dev`'s URL
shape changes. That's inherent to a manual, dev-time vendoring step (per the epic: "the sync step
is a build/dev-time fetch... not a runtime request") ‚Äî nothing to unit-test there, and adding CI
enforcement was never in this ticket's acceptance criterion.

## Open concerns / known limitations

1. **Network dependency at dev/CI-authoring time.** `just sync-kit` requires `b28.dev` to be
   reachable. It's a manual step (not part of `npm run build`), so this only affects whoever runs
   `sync-kit`, not every build/deploy. Confirmed live and stable during this session (fetched
   successfully 3 separate times: once in Research, twice in Implement).
2. **No content/shape validation beyond curl's HTTP-failure detection.** `curl -f` catches HTTP
   errors (4xx/5xx) but wouldn't catch, say, the upstream kit being replaced with a differently-
   shaped-but-200-OK file. Not flagged as a real risk ‚Äî this is a controlled internal source (the
   user's own `b28.dev`), and adding schema validation for a single CSS file is out of proportion
   to the ticket.
3. **The vendored file is now a tracked, hand-visible diff surface.** Every future `sync-kit`
   re-run will show as a full-file diff on any kit change (not a semantic diff) ‚Äî expected and
   matches the `wrangler.jsonc` precedent already in this repo; not a defect.
4. **No CI/build-time enforcement that the vendored copy hasn't silently drifted from upstream.**
   Out of scope per this ticket's acceptance criterion and the epic's own framing of sync as a
   manual step; flagging in case a later ticket wants to add a "sync-kit --check" mode.
