# T-009-01-01 — Design: vendor-b28-clay-kit-via-justfile

## Decisions to make

1. Where does the vendored file live?
2. What tool/command does the fetch?
3. How is the `justfile` recipe named/shaped, and does it need anything beyond the one recipe?
4. Is the vendored file committed to git or gitignored?
5. Any failure-mode handling (network error, empty response, etc.)?

## 1. Vendored path

**Options considered:**

- `styles/vendor/b28-clay.css` — the ticket's own suggested example path.
- `public/vendor/b28-clay.css` — Next.js convention for static assets served as-is.
- `app/vendor/b28-clay.css` — colocated with the App Router tree.

**Decision: `styles/vendor/b28-clay.css`.**

Rejected `public/`: `public/` is for browser-served static assets fetched at runtime
(images, favicons); this file is consumed at *build time* via a CSS `@import` from
`app/globals.css` (per T-009-01-02's acceptance criterion — "app/globals.css imports the vendored
kit file"), bundled by Vite/Tailwind like any other source CSS, not fetched by the client
separately. Putting it in `public/` would work mechanically (Next serves it) but signals the wrong
thing and adds an unnecessary second HTTP request in production.

Rejected `app/vendor/`: `app/` in the App Router is route/page tree — mixing a non-route vendor
asset into it is confusing, and this repo's own `CLAUDE.md` source layout table already separates
`app/` (entry), `components/`, `lib/` cleanly; a `styles/` top-level directory is a clean fourth
category for "things that are CSS, not components or routes."

`styles/vendor/b28-clay.css` also directly matches the ticket text and the global CLAUDE.md's
"vendoring... `just sync-kit`" language — using the exact suggested path removes any ambiguity for
the two downstream tickets that will reference it.

## 2. Fetch tool

**Decision: `curl -fsSL`.**

- `curl` is standard on macOS/Linux dev machines and CI images; no new dependency to install.
- Flags: `-f` (fail on HTTP error instead of writing an error-page body to the file), `-s`
  (silent, no progress meter cluttering `just` output), `-S` (still show errors despite `-s`),
  `-L` (follow redirects, cheap insurance if b28.dev ever redirects `/kit/*`).
- Alternative considered: a Node/`fetch` script. Rejected — adds a script file and an npm
  dependency surface for something `curl` does in one line; nothing else in this repo's tooling
  uses a Node fetch script for similar jobs (`docs/deploy.md`'s `wrangler.jsonc` generation is a
  vendor-tool CLI, not a hand-rolled fetch).

## 3. `justfile` shape

**Decision:** one `justfile` at repo root, one recipe:

```just
# Vendor the shared b28.dev clay kit into this repo.
sync-kit:
    mkdir -p styles/vendor
    curl -fsSL https://b28.dev/kit/b28-clay.css -o styles/vendor/b28-clay.css
    @echo "Synced styles/vendor/b28-clay.css"
```

Considered adding a `default` recipe that lists available recipes (`just --list` already does this
natively — a `default: @just --list` recipe is a common `just` convention for discoverability).
Included it since it's idiomatic, near-zero cost, and this is the *first* `justfile` in the repo —
worth making `just` (bare) self-documenting rather than erroring with "no default recipe" the
first time someone runs it.

Rejected: parameterizing the recipe (e.g. `sync-kit url=...`) — YAGNI. There is exactly one kit
URL, defined by the epic and the global CLAUDE.md as the fixed source of truth; a parameter adds
surface area nothing in this epic needs.

Rejected: making `sync-kit` idempotent-silent-skip (e.g. skip fetch if file exists) — the whole
point of the recipe per the epic's "Done looks like" section is "running `just sync-kit` followed
by a rebuild picks up a kit palette change" — it must always re-fetch and overwrite, never skip.

## 4. Git tracking

**Decision: commit the vendored file (not gitignored).**

Confirmed by research: this mirrors the existing `wrangler.jsonc` precedent (generated by a tool,
committed verbatim, reviewed as a diff). It's also required for the file to be *importable* by
`app/globals.css` in a checked-out repo without every developer/CI runner needing network access
to `b28.dev` at build time — `T-009-01-02`'s build (`npm run build`) must succeed from a clean
checkout. `.gitignore` is left untouched — nothing currently excludes `styles/`.

## 5. Failure handling

**Decision: rely on `curl -f` + `just`'s default fail-fast behavior; no custom error handling.**

`just` recipes run under `set -e`-like semantics by default (a failing line aborts the recipe with
a non-zero exit and visible error). `curl -f` turns an HTTP error response into a non-zero curl
exit code instead of silently writing a 404 page to disk. That's sufficient: no legal-placement-
style "empty result is a valid outcome" case here (unlike `lib/bot.ts`'s `chooseMove`) — a failed
fetch is always an error, not a state to handle gracefully. No retry logic — this is a manual
dev-time command, not a build-pipeline step with uptime requirements (confirmed out of scope per
E-009: "the sync step is a build/dev-time fetch... not a runtime request").

## What's explicitly out of scope here (owned by later tickets)

- Importing the vendored file from `app/globals.css` (`T-009-01-02`).
- Mapping `--clay-*` tokens into Tailwind `@theme` (`T-009-01-02`).
- Loading Lora/Karla fonts (`T-009-01-03`).
- Any component restyling (later stories in E-009).
- Wiring `sync-kit` into `npm run build` / CI — not asked for by the acceptance criterion, and the
  epic frames sync as a manual dev-time step, not a build dependency.

## Verification approach

Run `just sync-kit` for real during Implement and confirm `styles/vendor/b28-clay.css` exists on
disk with the real kit content (matches the 5744-byte fetch already done during Research) — this
satisfies the acceptance criterion directly and is cheap since the URL is confirmed live.
