# Deploying webapp-demo-lisa-vend to Cloudflare Workers

This project builds and deploys with **vinext** (Cloudflare's Vite-based, Next.js-API-compatible
runtime) and ships to **Cloudflare Workers**. The committed deploy artifact is
[`wrangler.jsonc`](../wrangler.jsonc) at the repo root.

> **Out of scope for T-006-03-02.** This ticket commits the deploy artifact and documents the
> authenticated deploy path. The **live, authenticated deploy** and the resulting
> `*.workers.dev` **URL smoke-test are intentionally NOT performed here** — they require a
> Cloudflare login and are the manual follow-up described below.

## What `wrangler.jsonc` declares

`wrangler.jsonc` is generated by vinext (see "How it was generated") and committed verbatim. Its
fields:

| Field | Value | Meaning |
|---|---|---|
| `name` | `webapp-demo-lisa-vend` | Worker name (from `package.json`). Becomes the `*.workers.dev` subdomain. |
| `compatibility_date` | `2026-07-01` | Cloudflare Workers runtime compatibility date. |
| `compatibility_flags` | `["nodejs_compat"]` | Enables Node.js built-ins the vinext server handler needs. |
| `main` | `vinext/server/fetch-handler` | The Worker entry — vinext's built-in fetch handler (this repo has no custom `worker/` entry). |
| `assets.directory` | `dist/client` | Static assets served from vinext's client build output. |
| `assets.binding` | `ASSETS` | Binding name the Worker uses to serve those assets. |

There are **no `kv_namespaces` or `images` bindings**: this is a pure client-side Tetris game (no
`next/image`, no route handlers, no server actions, no data cache), so there is nothing to
provision. The artifact is self-contained — no placeholder KV namespace id to fill in.

## Prerequisites (for a real deploy)

1. A **Cloudflare account**.
2. **`wrangler` authenticated** — either `npx wrangler login` (interactive) or a
   `CLOUDFLARE_API_TOKEN` environment variable with Workers deploy permission.

## One-time deploy wiring (login-gated follow-up — not done in this ticket)

The committed `wrangler.jsonc` is the declarative artifact, but a real `vinext deploy` also needs
the Cloudflare build wiring (a cloudflare plugin in `vite.config.ts` and the deploy dependencies).
Add it with vinext's own initializer, matching the options this artifact was generated with:

```bash
npx vinext init --platform=cloudflare --data-cache=none --image-optimization=none
```

This:
- installs `@cloudflare/vite-plugin`, `@vinext/cloudflare`, and `wrangler`;
- injects the Cloudflare plugin into `vite.config.ts`;
- **detects the existing `wrangler.jsonc` and does not overwrite it.**

> In a non-interactive/agent environment, the `--platform`, `--data-cache`, and
> `--image-optimization` flags are **required** (vinext refuses to prompt).

## Deploy

Once the wiring above is in place and `wrangler` is authenticated:

```bash
npx @vinext/cloudflare deploy          # canonical: builds with Vite, deploys via wrangler
npx @vinext/cloudflare deploy --dry-run # validate setup only — no build, no deploy
```

Notes:
- `vinext deploy` is a **deprecated alias** that prints a warning and forwards to
  `@vinext/cloudflare`; it will be removed in a future release. Prefer the `npx @vinext/cloudflare
  deploy` form above.
- `--dry-run` validates that the wrangler file, the deploy dependencies, and the cloudflare vite
  plugin are all present; it will error with "Run `vinext init --platform=cloudflare` first" if the
  wiring step above has not been run.

## How `wrangler.jsonc` was generated

It is the byte-for-byte output of vinext's config generator for this repo:

```
vinext init --platform=cloudflare --data-cache=none --image-optimization=none
```

(equivalently, `generateWranglerConfig` in `vinext/dist/init-cloudflare.js`). The committed file
is not hand-authored — re-running the generator produces an identical file, so it can be
regenerated on a vinext upgrade and diffed.

## After a real deploy (out of scope here)

The deploy prints a `https://webapp-demo-lisa-vend.<subdomain>.workers.dev` URL. Smoke-testing that
URL (load the page, play a line, confirm the neon/glass board renders) is the manual verification
step — deferred, per this ticket's scope.
